漏洞成因简要分析：

OpenFaiMessage将从收件箱直接读取配置并返回。由于任意用户均可对自身配置进行任意修改，这里是一个可控输入，随后DeserializeModelFAI方法将在没有任何SerializationBinder的情况下直接进行反序列化。SerializationBinder的作用是控制反序列化过程中使用的类，如果使用SerializationBinder对使用的类进行了限制，则无法使用相关的反序列化链

CVE-2020-16875

漏洞成因简要分析

在https://ip/ecp/DLPPolicy/ManagePolicyFromISV.aspx处exchange提供了一个上传xml的功能，用户可以上传xml以实现dlp（Data loss Prevention）。dlp主要用于防止数据泄露，它可以通过关键字匹配，正则表达式，词典匹配等方式分析内容是否为敏感内容。
该漏洞的出现是由于后台对于上传的xml的commandBlock参数没有进行严格的限制导致攻击者可以通过上传恶意的xml进行命令执行

CVE-2020-0688

漏洞成因简要分析

在exchange server安装后validationKey和decryptionKey是默认的，而这两个参数又是用来保护ViewState的。ViewState是服务端存放在客户端的数据，他是以序列化格式存储的。所以我们可以通过默认的validationKey和decryptionKey控制ViewState以达到反序列化执行命令的目的

影响版本

Microsoft Exchange Server 2010 Service Pack 3

Microsoft Exchange Server 2013

Microsoft Exchange Server 2016

Microsoft Exchange Server 2019

漏洞利用

利用此漏洞需要一下几个参数：

  • validationkey 默认为CB2721ABDAF8E9DC516D621D8B8BF13A2C9E8689A25303BF
  • validationalg = SHA1（默认）
  • generator=B97B4E27（大部分情况下默认）
  • viewstateuserkey = ASP.NET_SessionId

所以我们需要手工获取的是后两个参数，主要是ASP.NET_SessionId
ASP.NET_SessionId可以在/ecp/default.aspx的cookie中获取

generator可以在/ecp/default.aspx的response中搜索__VIEWSTATEGENERATOR获取

CVE-2018-8581

漏洞成因简要分析

该漏洞允许任何已经登录的用户向exchange上的任何用户发送任意链接作为推送，服务器将会访问此链接。如果将此链接设置为恶意的payload，则可以获取到exchange的机器账户的net-ntlm值。获取到此net-ntlm值之后又两种利用方式：

  1. 将此net-ntlm哈希中继回exchange服务器的ews服务，构造恶意soap头，通过sid伪造任意用户的身份，进行收件，发件等操作

  2. 由于再默认情况下exchange的机器用户具有WriteACL的权限，所以可以将此net-ntlm哈希中继到dc的ldap上，修改ACL为普通域内用户添加权限，使得普通用户也可以执行DCSync。
    DCSync是域中用于不同DC之间同步数据的一种方式，具有特定权限的用户可以被伪造为恶意dc向原DC发送数据同步请求从而获取域内所有成员的hash。

影响版本

Exchange Server 2010

Exchange Server 2013

Exchange Server 2016

漏洞利用：

首先设置ntlmrelay中继（需要用linux主机）

之后使用exp发送恶意链接

最后利用dcsync到处域内成员hash（需要用Linux主机）
